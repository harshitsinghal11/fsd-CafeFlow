## Project Vision
CafeFlow is a cafe ordering system for a single cafe setup. It lets customers browse menu categories, place an order without creating an account, receive a token number, and track orders by phone number. It also gives cafe staff a protected admin dashboard to manage kitchen order flow.

## Problem Statement
Manual cafe ordering creates queue delays, unclear kitchen handoff, and poor order visibility for both customers and staff. The project solves this by moving ordering, token generation, order tracking, and kitchen status updates into one web application.

## Goals & Objectives
- Let customers browse the menu by category.
- Let customers add items to a cart and place an order quickly.
- Generate a token number for each order.
- Let customers check their past and current orders using a phone number.
- Let staff view active orders and update order status.
- Show simple weekly sales analytics for completed orders.

## Target Audience
- Cafe customers placing orders on-site or nearby.
- Cafe staff managing preparation and completion.
- Cafe owner or manager checking weekly order performance.

## User Roles & Permissions
- Guest Customer
  Can browse menu pages, add items to cart, place orders, and look up orders by phone number.
- Admin Staff
  Can log in with a PIN, access `/admin` routes, read all orders, update order status, and view weekly analytics.
- Additional roles
  Project not supported.

## Core Features
- Landing page with menu category navigation.
- Category pages for Cold Coffee, Thick Shakes, and Mocktails.
- Persistent cart using local storage.
- Cart expiry timer set to 10 minutes.
- Checkout with customer name and 10-digit phone number.
- Token-based order confirmation using `order_no`.
- My Orders page with phone-based lookup.
- Admin login with signed session cookie.
- Admin dashboard for active and historical orders.
- Weekly analytics for completed orders.

## Functional Requirements
- The system must show three customer menu categories.
- The system must fetch menu items from Supabase by category.
- The cart must support add, remove, increase, decrease, clear, and total calculations.
- The cart must expire automatically after 10 minutes of inactivity from first item add.
- Checkout must reject empty carts, blank names, and invalid phone numbers.
- Order placement must create a new `orders` row with `pending` status.
- The system must return the generated `order_no` after successful checkout.
- Customers must be able to look up orders using a 10-digit phone number.
- Admin users must log in with the configured PIN.
- Admin routes except `/admin/login` must require a valid session cookie.
- Admin users must be able to move orders between `pending`, `preparing`, `completed`, and `cancelled`.
- Admin analytics must summarize completed orders for the current week.

## Non-Functional Requirements
- The application must work as a responsive web app on mobile and desktop.
- Customer and admin flows must return clear validation errors for bad input.
- Sensitive admin data access must stay server-side.
- Admin session cookies must be `httpOnly` and signed.
- Cart state must persist across refreshes in the browser.
- Admin order data should refresh automatically through polling.

## Business Rules
- Customer phone number must contain exactly 10 digits.
- Customer name must not be blank.
- Orders start with status `pending`.
- Valid order statuses are `pending`, `preparing`, `completed`, and `cancelled`.
- `order_no` is generated by the database and starts from 100.
- Customers can place orders without authentication.
- Only admins with a valid session can read or update all orders.
- The public menu only shows items marked available through RLS.
- Weekly analytics count only completed orders.

## Assumptions
- The project serves one cafe location.
- Menu data is seeded manually in Supabase.
- Currency is Indian Rupees.
- One shared admin PIN is acceptable for staff access.
- Customers identify their orders only through phone number lookup.

## Constraints
- The project depends on Supabase environment variables being configured.
- The menu structure is limited to the three hardcoded category routes present in the app.
- There is no payment gateway integration.
- There is no customer account system.
- There is no multi-branch or franchise support.
